# ADR-0001 — Trinity Pattern — REST API + Library + MCP Server

> **Architectural Decision Record.** Registra **uma** decisão arquitetural não-óbvia,

## 1. Status

`proposto` — desde `2026-05-09`.

---

## 2. Contexto

Serviços da plataforma historicamente eram monolíticos (apenas REST API). Agentes de IA precisam acessar features de forma estruturada via MCP. Aplicar padrão consistente garante cobertura uniforme, testabilidade, e evolução controlada.

---

## 3. Decisão

Vamos adotar Trinity Pattern: cada serviço expõe (1) REST API (humanos/sistemas), (2) Python library (outras services), (3) MCP Server (agentes IA). Começar pelos 5 core services (auth, admin, governance, scheduler, connectors) + docs-mcp como infraestrutura. Padrão: mcp/ directory no repo, pyproject.toml, src/server/mcp_server.py, pytest suite completa.

## 5. Consequências

✅ Agentes IA ganham cobertura uniforme de features. ✅ Testes validam contract de cada tool. ✅ Reutilizável: bibliotecas private importáveis por serviços. ✅ Evolutivo: novos tools adicionados sem breaking change. ⚠️ Requer manutenção de 3 interfaces (REST/lib/MCP). ⚠️ Docs das tools deve estar sincronizada com REST docs.
